질문의 내용이 'f' 옵션의 차이라고 생각하고 답변 드릴게요.

  브랜치는 커밋이 다른 방향으로 나가게 하는 한 지점이라고 생각하시면 됩니다. '(나무의) 가지'로 해석되지만 가지의 시작점 개념으로 생각하시는 것이 이해하는 데 도움이 될 것입니다.

  실제로 'f' 옵션은 사용해 보지 않았지만 'throw away local modifications' 을 설명하자면 몇 가지 다른 개념에 대해서 이해 하셔야 합니다.

로컬 변경 사항들은 워킹 트리에서 언트랙티드 파일과 트랙티드 파일 중 변경된 파일, (테스트는 안해보았지만) 스테이지드에 있는 파일 까지 포함 될 겁니다. 

가끔 checkout 을 하려고 할 때, 로컬 변경 사항 때문에 체크 아웃이 안되는 경우가 있습니다. 저는 해당 상황을 경험해 보지는 못했지만 (로컬 변경 사항과 변경 하려는 브랜치의 차이 때문에 그럴 것 같습니다..ㅎ), 그럴 경우 강제로 변경사항을 무시하고 체크아웃을 하도록 돕는 옵션입니다.(help 설명)



1.워킹트리(워킹 카피), 

1.1. 언트랙티드 : 변경사항 중 git 이 신경 쓰지 않는 부분, add 를 통해서 트랙티드로 될 수 있음. 

1.2. 트랙티드  : git 이 변경사항을 신경 쓰는 부분, commit 를 통해서 변경 사항을 저장할 수 있음.

1.3. 이그노어드 : git 에서 무시되는 파일.

2. 인덱스

2.1 스테이지드 : 변경 사항이 커밋 되기 전에 잠시 머무는 공간

2.2 커밋티드 : 변경이 저장되는 곳, 히스토리로 남는 공간.
